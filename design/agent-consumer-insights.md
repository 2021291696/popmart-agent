# 2.4 消费者洞察Agent（RAG知识库）

> 🏆 **P0落地Agent** — 节点3将完整实现此Agent

## 为什么做这个

泡泡玛特有海量消费者数据散落在不同系统——产品经理想快速了解"Molly和Dimio的消费者画像有什么不同"，需要跨系统拉数据+找分析师跑SQL，两天后才有答案。消费者洞察Agent用RAG知识库+智能路由，把"两天"变成"两分钟"，并且每个回答都带引用来源和置信度。

## Agent职责声明（System Prompt草案）

```
你是泡泡玛特消费者洞察Agent。你基于知识库中的泡泡玛特产品信息、
财报数据、市场分析和用户评论来回答问题。

## 核心规则
1. **必须基于资料回答**：你的知识来源是RAG检索到的文档段落。如果资料不足以回答问题，明确说明"目前知识库中缺少以下信息：..."
2. **必须引用来源**：每个回答必须标注信息来源（文档ID+段落编号）
3. **必须标注置信度**：确定(>90%) / 较确定(70-90%) / 不确定(<70%)
4. **不要编造数据**：如果知识库中没有具体数字，不要凭空给出数字

## 工具
- rag_query: 查询知识库（产品/财报/市场数据）
- user_profile_build: 生成某IP的典型消费者画像
- compare_ips: 两个IP的消费者差异对比
- review_mining: 用户评论挖掘（高频词/情感趋势）
```

## 工具清单（4个MCP工具）

| 工具 | 核心逻辑 | tool_description亮点 |
|------|---------|---------------------|
| rag_query | Embedding→ChromaDB检索→LLM基于检索结果生成 | "查询泡泡玛特知识库。适用于产品信息、历史数据、财报分析。注意：不适用于实时市场动态——实时信息请用web_search" |
| user_profile_build | 检索该IP的历史销售数据+评论→生成画像 | "输入IP名称，输出消费者画像：年龄/性别/地域/购买频次/偏好款式/价格敏感度" |
| compare_ips | 对两个IP分别检索→差异对比 | "输入两个IP名称，输出差异对比报告：消费者重叠度/价格带差异/地域偏好差异/增长趋势对比" |
| review_mining | 检索指定时间段的评论→高频词+情感趋势 | "输入时间范围，输出：高频关键词Top20/情感趋势曲线/正面和负面提及最多的产品特征" |

## RAG流水线设计

```
数据源（三层）          → 预处理             → Embedding         → 存储          → 检索              → 生成
──────────────────────────────────────────────────────────────────────────────────────────────────
年报/财报PDF            chunk_size=512      bge-large-zh       ChromaDB       similarity(K=3)    SystemPrompt
产品目录(官网)          overlap=64          本地HuggingFace    Collection:     - 事实查询          + 检索结果
二手市场公开数据        去重+清洗+标注来源    免费+中文最优       popmart_docs    MMR(K=5)          + LLM
用户评论(公开)                                              popmart_products  - 分析问题          → 带来源的回答
                                                           popmart_market                        + 置信度
```

### 切分策略：chunk_size=512, overlap=64

**为什么选512而不是1000？**
泡泡玛特的产品描述大多是短段落（款式名/艺术家/系列背景），512能在"一个chunk包含完整的产品信息"和"检索足够精准"之间取得平衡。太大了→检索混入无关信息；太小了→一个产品信息被拆成两段→信息断裂。

**为什么overlap=64？** 防止关键信息恰好落在两个chunk的边界上。64个token的overlap=约40-50个中文字符。

### Embedding：bge-large-zh

在C-MTEB中文基准上比OpenAI text-embedding-3-small高约3个百分点。本地部署零费用。生产环境如需更高精度可升级为bge-m3（支持多语言）。

### 检索策略：双模式

| 问题类型 | 检索方式 | K值 | 理由 |
|---------|:--:|:--:|------|
| 事实查询（"Dimio有哪些款式"） | 相似度排序 | 3 | 需要精确匹配，K=3够用 |
| 分析问题（"Molly和Dimio的消费者有什么不同"） | MMR | 5 | 需要多样性，MMR去重避免返回5段差不多的内容 |

### 路由策略（三层规则路由）

```
用户问题
  │
  ├── 含"产品/款式/系列/IP名"等关键词 → RAG知识库（rag_query）
  │
  ├── 含"最近/现在/当前/今天"等时间词 → 转IP情报Agent（外部搜索）
  │
  └── 综合分析（"为什么/对比/趋势/原因"）→ RAG + IP情报 双路合并
```

**为什么用规则路由而非让LLM判断？**
- 规则路由更快（零额外LLM调用）
- 更可控（不会"判断错了去错数据源"）
- 面试能讲：这是"什么时候不该让LLM决策"的工程判断

## 评估体系

eval_rag.py — 20个测试问题（事实类10+分析类10），跑Agent→自动评分：

| 评分 | 标准 | 示例 |
|:--:|------|------|
| 2分 | 完全正确，引用准确 | Q:"Dimio诞生于哪年" A:"2019年"[来源:popmart_products#42] |
| 1分 | 部分正确，漏了部分信息 | 回答了年份但没回答设计师名字 |
| 0分 | 错误 | 把MOLLY的诞生年份说成DIMOO的 |
| -1分 | 幻觉（知识库没有但编了） | 知识库没有的数据却提供了"具体数字" |

输出metrics.json：准确率/召回率/幻觉率/工具调用有效率。

## 关键决策

1. **Embedding选bge-large-zh并非OpenAI** — 中文基准高3%+免费。面试能讲"我知道怎么评估Embedding模型——看MTEB基准+在你具体数据上的召回率"

2. **路由用规则而非LLM** — 快、可控、省token。面试能讲"不是所有环节都需要LLM的智能——有时规则的确定性比LLM的灵活性更有价值"

3. **评估体系是最大差异化武器** — 大部分实习生候选人做RAG但没做评估。你拿出metrics.json说"准确率XX%，幻觉率XX%"，面试官眼睛会亮。

## 面试能讲什么

> "消费者洞察Agent的核心是三重设计：RAG流水线+智能路由+量化评估。
>
> RAG流水线方面，chunk_size我选了512——因为泡泡玛特的产品描述是短段落，512精准匹配。
> Embedding选了bge-large-zh——在C-MTEB中文基准上比OpenAI高3%，且本地部署零成本。
>
> 路由方面我用了规则触发——产品事实走RAG、实时动态转IP情报Agent、综合分析双路合并。
> 为什么不用LLM判断路由？因为两条路的选择不需要'智能'——它有明确的规则。规则的确定性在这种场景下比LLM的灵活性更有价值。
>
> 但最核心的是——我建立了一套评估体系。20个测试问题，4个评分等级，输出准确率/召回率/幻觉率。
> [打开metrics.json] 我的准确率是XX%，幻觉率是XX%。我知道这个数字还不够好，
> 下一步会从chunk策略/Embedding选型/检索算法/Prompt优化/数据质量五个方向迭代。
>
> 评估体系是我和大多数候选人拉开差距的地方——他们做了RAG，我评估了RAG。"
